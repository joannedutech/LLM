## 什麼是 User Story Mapping？

User Story Mapping 是 Jeff Patton 提出的需求整理方法。它的核心概念很簡單：**把用戶故事按照使用流程排列，再按照優先級分層**，就能產生一張二維的需求地圖。

![團隊協作](https://images.unsplash.com/photo-1542626991-cbc4e32524cc?w=800)

橫軸是**使用者旅程**（用戶從頭到尾會經歷的步驟），縱軸是**優先級**（從上到下越來越不重要）。這張地圖能讓團隊對「我們在做什麼」和「先做什麼」有共同的視覺化理解。

---

## 為什麼 Flat Backlog 不夠？

傳統的 backlog 就是一個排序過的清單。問題是：

1. **失去全貌**：你知道第 1 名是什麼，但不知道整個產品的輪廓
2. **容易偏科**：某個功能模組的故事特別多，就佔據了 backlog 的前幾名，其他模組被忽略
3. **難以切版本**：不知道哪些故事組合在一起才能構成一個有意義的 release

Story Map 解決了這些問題——它讓你同時看見「廣度」和「深度」。

## 實作步驟

### Step 1：定義用戶活動（Activities）

先列出用戶的主要活動，這是地圖最上層的「骨幹」。

以一個電商網站為例：

```
瀏覽商品 → 搜尋商品 → 查看商品 → 加入購物車 → 結帳 → 追蹤訂單
```

### Step 2：拆解用戶任務（Tasks）

每個活動下面，列出用戶會執行的具體任務。

```
搜尋商品
├── 輸入關鍵字搜尋
├── 按分類瀏覽
├── 使用篩選條件（價格、品牌）
└── 查看搜尋建議
```

### Step 3：加入用戶故事（Stories）

每個任務下面，展開具體的用戶故事，並按優先級由上到下排列。

![Story Map 結構](https://images.unsplash.com/photo-1611224923853-80b023f02d71?w=800)

### Step 4：畫出版本切割線

在地圖上畫水平線，線以上的故事構成一個版本：

- **Release 1（MVP）**：每個活動只取最上面 1-2 個故事，確保用戶能走完整個流程
- **Release 2**：補充重要的功能細節
- **Release 3**：加入進階功能和優化

## MVP 切割的關鍵原則

> MVP 不是「功能最少的產品」，而是「能完成整個用戶旅程的最小版本」。

這是 Story Map 最強大的地方——它幫你確保 MVP **橫向完整**。很多團隊的 MVP 錯誤是把某一個模組做得很深，但其他模組完全空白，用戶根本走不完流程。

### 好的 MVP 切法

| 活動 | MVP 包含 | MVP 不包含 |
|------|----------|------------|
| 搜尋商品 | 關鍵字搜尋 | 篩選條件、搜尋建議 |
| 查看商品 | 基本資訊、圖片 | 評價、推薦商品 |
| 結帳 | 信用卡付款 | 超商取貨、分期付款 |
| 追蹤訂單 | 訂單狀態查詢 | 即時物流追蹤 |

每個環節都有，但每個環節都只做最基本的。

## 工作坊怎麼跑？

一個完整的 Story Mapping 工作坊通常需要 2-4 小時：

| 階段 | 時間 | 參與者 | 產出 |
|------|------|--------|------|
| 定義用戶角色 | 20 min | PM + 設計 + 工程 | Persona 卡片 |
| 列出活動與任務 | 40 min | 全員 | 地圖骨幹 |
| 展開用戶故事 | 60 min | 全員 | 完整地圖 |
| 排序與切版本 | 40 min | PM 主導 | Release 計畫 |

### 實用提示

- **用實體便利貼**（或 Miro/FigJam）：視覺化很重要，不要用試算表
- **邀請工程師參與**：他們能即時回饋技術可行性和工作量
- **先發散再收斂**：前半段不要急著排優先級，先把所有故事寫出來
- **拍照存檔**：工作坊結束後立刻拍照，再轉成數位版本

Story Mapping 不只是一個工具，它是一個**對話的框架**。最大的價值不在於產出的那張地圖，而在於過程中團隊建立的共同理解。
